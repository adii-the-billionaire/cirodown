2. Design goals | â‡‘ toc | â†‘ parent "Cirodown"

Cirodown is designed entirely to allow writing complex professional HTML and PDF scientific books, blogs, articles and encyclopedias. (ğŸ”— link)

Cirodown aims to be the ultimate LaTeX "killer", allowing books to be finally published as either HTML or PDF painlessly (LaTeX being only a backend to PDF generation), supporting out of the box feature such as: (ğŸ”— link)

  • references to headers, images, etc. with error checking: never break internal links gain
  • KaTeX server side Mathematics
  • multi file features out of the box so you don't need a separate wrapper like Jekyll to make a multi-page website:
    • Internal cross file references
    • table of contents that crosses input files
    • inotifywait watch rebuild server
    • cross file includes that work as links on multi output file mode, and true includes in single output file mode
    • cross file configuration files to factor out common page parts like headers and footers

It is meant to be both saner and more powerful than Markdown and Asciidoctor. (ğŸ”— link)

The tradeoff for those features is that the language is slightly heavier to read and write. (ğŸ”— link)

It is intended that this will be an acceptable downside as Cirodown will be used primarily large complex content such as books rather than forum posts, and will therefore primarily written either: (ğŸ”— link)

  • in text editors locally, where users have more features than in random browser textareas
  • in a dedicated website that will revolutionize education, and therefore have a good JavaScript editing interface: https://github.com/cirosantilli/write-free-science-books-to-get-famous-website

2.1. Saner | â‡‘ toc | â†‘ parent "Design goals"

To be saner than both Markdown and Asciidoctor, Cirodown has exactly five magic characters, with similar functions as in LaTeX: (ğŸ”— link)

  • \ backslash to start a macro, like LaTeX
  • { and }: left and right square brackets to delimit optional macro arguments
  • [ and ]: left and right curly braces bracket to start an optional arguments

And double blank newlines for paragraphs if you are pedantic. (ğŸ”— link)

We would like to have only square brackets for both optional and mandatory to have even less magic characters, but that would make the language difficult to parse for computer and humans. LaTeX was right for once! (ğŸ”— link)

This produces a very regular syntax that is easy to learn, including doing: (ğŸ”— link)

  • arbitrary nesting of elements
  • adding arbitrary properties to elements

This sanity also makes the end tail learning curve of the endless edge cases found in Markdown and Asciidoctor disappear. (ğŸ”— link)

The language is designed to be philosophically isomorphic to HTML to: (ğŸ”— link)

  • further reduce the learning curve
  • ensure that most of HTML constructs can be reached, including arbitrary nesting

More precisely: (ğŸ”— link)

  • macro names map to tag names, e.g.: \a to <a

  • one of the arguments of macros, maps to the content of the HTML element, and the others map to attributes. (ğŸ”— link)

    E.g., in a link: (ğŸ”— link) 

    \a[http://example.com][Link text]
    (ğŸ”— link)

    the first macro argument: (ğŸ”— link) 

    http://example.com
    (ğŸ”— link)

    maps to the href of <a, and the second macro argument: (ğŸ”— link) 

    Link text
    (ğŸ”— link)

    maps to the internal content of <a>Link text<> (ğŸ”— link)

2.2. More powerful | â‡‘ toc | â†‘ parent "Design goals"

The high sanity of Cirodown, also makes creating new macro extensions extremely easy and intuitive. (ğŸ”— link)

All built-in language features use the exact same API as new extensions, which ensures that the extension API is sane forever. (ğŸ”— link)

Markdown is clearly missing many key features such as block attributes and Internal cross references \x, and has no standardized extension mechanism. (ğŸ”— link)

The "more powerful than Asciidoctor" part is only partially true, since Asciidoctor is very featureful can do basically anything through extensions. (ğŸ”— link)

The difference is mostly that Cirodown is completely and entirely focused on making amazing scientific books, and so will have key features for that application out-of-the box, notably: (ğŸ”— link)

  • amazing header/ToC/ID features including proper error reports: never have a internal broken link or duplicate ID again
  • Mathematics \M and \m
  • --publish: we take care of website publishing for you out-of-the-bos, no need for extra stuff like Jekyll mess

Another advantage over Asciidoctor is that the reference implementation of Cirodown is in JavaScript, and can therefore be used on browser live preview out of the box. Asciidoctor does Transpile to JS with Opal, but who wants to deal with that layer of complexity? (ğŸ”— link)

5.3.1. Internal cross file references internals | â‡‘ toc | â†‘ parent "Internal cross file references"

When running in Node.js, Cirodown dumps the IDs of all processed files to a out/ids.sqlite3 file, and then reads from that file when IDs are needed. (ğŸ”— link)

When converting a directory, out/ids.sqlite3 is placed inside the input directory. When converting individual files, out/ids.sqlite3 is placed in the current working directory. The file is not created or used when handling input from stdin. (ğŸ”— link)

When running in the browser, the same JavaScript API will send queries to the server instead of a local SQLite database. (ğŸ”— link)

To inspect the the ID database to debug it, you can use: (ğŸ”— link) 

sqlite3 out/db.sqlite3 .dump

5.4. Automatic ID from title | â‡‘ toc | â†‘ parent "Internal cross references \x"

6.4. Includes \include | â‡‘ toc | â†‘ parent "Headers \h"

The \include macro allows including an external Cirodown headers under the current header. (ğŸ”— link)

It exists to allow optional single page HTML output while still retaining the ability to: (ğŸ”— link)

  • split up large input files into multiple files to make renders faster during document development
  • suggest an optional custom output split with one HTML output per Cirodown input, in order to avoid extremely large HTML pages which could be slow to load

\include takes one mandatory argument, which is the relative path without extension to the .ciro file that you want to include. (ğŸ”— link)

Things would have been a bit more clean if the argument would take the ID of the header you want to include rather than the filename, which is analogous to how \x works, but that would mean that a single page build would require an initial parse to determine IDs, so we are just going with the easier option of pointing out the file name directly. (ğŸ”— link)

Headers of the included document are automatically shifted to match the level of the child of the level where they are being included. (ğŸ”— link)

If --html-single-page is given, the external document is rendered embedded into the current document directly, essentially as if the source had been copy pasted (except for small corrections such as the header offsets). (ğŸ”— link)

Otherwise, the following effects happen: (ğŸ”— link)

  • The headers of the included tree appear in the table of contents of the document as links to the corresponding external files. (ğŸ”— link)

    This is implemented simply by reading a previously generated database file much like Internal cross file references internals, which avoids the slowdown of parsing all included files every time. (ğŸ”— link)

    As a result, you have to do an initial parse of all files in the project to extract their headers however, just as you would need to do when linking to those headers. (ğŸ”— link)

  • the include itself renders as a link to the included document
  • --html-single-page

8.1. Where to store images | â‡‘ toc | â†‘ parent "Images \image"

If you are making a limited repository that will not have a ton of images, then you can get away with simply git tracking your images in the main repository. (ğŸ”— link)

With this setup, no further action is needed. For example, with a file structure of: (ğŸ”— link)

just use the image from

as: (ğŸ”— link)

However, if you are making a huge tutorial, which can have a huge undefined number of images (i.e. any scientific book), then you likely don't want to git track your images in the git repository. (ğŸ”— link)

We recommend the following approach instead. (ğŸ”— link)

Create a separate GitHub repository in addition to the main one containing the text, for example: (ğŸ”— link)

  • ./my-tutorial/
  • ./my-tutorial-media/

The name *-media suffix is not mandatory, but if you use this default, cirodown will handle it for you without any further configuration. (ğŸ”— link)

And then set the media_source_type option in your cirodown.json: (ğŸ”— link) 

{
  media_source_type = "media_repo"
}

Now, simply drop your images into the my-tutorial-media repository and use them just as before, and everything will just work! (ğŸ”— link) 

\Image[Image_with_a_descriptive_title.jpeg]

cirodown will even automatically add and push used images in the my-tutorial-media repository for you during publishing! (ğŸ”— link)

You should then use the following rules inside my-tutorial-media: (ğŸ”— link)

  • give every file a very descriptive and unique name as a full English sentence
  • never ever delete any files, nor change their content, unless it is an improvement in format that does change the information contained of the image TODO link to nice Wikimedia Commons guideline page

This way, even though the repositories are not fully in sync, anyone who clones the latest version of the *-media directory will be able to view any version of the main repository. (ğŸ”— link)

Then, if one day the media repository ever blows up GitHub's limit, you can just migrate the images to another image server that allows arbitrary basenames, e.g. AWS, and just configure your project to use that new media base URL with the media_base_url option. (ğŸ”— link)

The reason why images should be kept in a separate repository is that images are hundreds or thousands of times larger than hand written text. (ğŸ”— link)

Therefore, images could easily fill up the maximum repository size you are allowed: https://webapps.stackexchange.com/questions/45254/file-size-and-storage-limits-on-github#84746 and then what will you do when GitHub comes asking you to reduce the repository size? (ğŸ”— link)

Git LFS is one approach to deal with this, but we feel that it adds too much development overhead. (ğŸ”— link)

13.1.1. Positional vs named arguments | â‡‘ toc | â†‘ parent "Macro argument syntax"

Every argument in Cirodown is either positional or named. (ğŸ”— link)

For example, in a header definition with an ID: (ğŸ”— link) 

\h[1][My asdf]{id=asdf qwer}{id_scope=myscope}

we have: (ğŸ”— link)

  • two positional argument: 1 and My asdf. Those are surrounded by and have no name

  • two named arguments: {id=asdf qwer} and {id_scope=myscope}. (ğŸ”— link)

    The first one has name id and the mandatory separator =, followed by the value asdf qwer. (ğŸ”— link)

You can determine if a macro is positional or named by using --help-macros. Its output contains something like: (ğŸ”— link) 

  "h": {
    "name": "h",
    "positional_args": [
      {
        "name": "level"
      },
      {
        "name": "content"
      }
    ],
    "named_args": {
      "id": {
        "name": "id"
      }
      "id_scope": {
        "name": "id_scope"
      }
    },

and so we see that level and content are positional arguments, and id and id_scope are named arguments. (ğŸ”— link)

Generally, positional arguments are few (otherwise it would be hard to know which is which is which), and are almost always used for a given element so tha they save us from typing the name too many times. (ğŸ”— link)

The order of positional arguments must of course be fixed, but named arguments can go anywhere. We can even mix positional and named arguments however we want, although this is not advised for clarity. (ğŸ”— link)

The following are therefore all equivalent: (ğŸ”— link) 

\h[1][My asdf]{id=asdf qwer}{id_scope=myscope}
\h[1][My asdf]{id_scope=myscope}{id=asdf qwer}
\h{id=asdf qwer}{id_scope=myscope}[1][My asdf]
\h{id_scope=myscope}[1]{id=asdf qwer}[My asdf]

Just like named arguments, positional arguments are never mandatory. (ğŸ”— link)

If not given, most positional arguments will default to an empty string. (ğŸ”— link)

However, some positional arguments can have special effects if not given. (ğŸ”— link)

For example, an anchor with the first positional argument present (the URL), but not the second positional argument (the link text) as in: (ğŸ”— link) 

\a[http://example.com]

has the special effect of generating automatic links as in: (ğŸ”— link) 

\a[http://example.com][http://example.com]

See also: Links \a. (ğŸ”— link)

13.1.5. Literal arguments | â‡‘ toc | â†‘ parent "Macro argument syntax"

Arguments that are opened with more than one square brackets [ or curly braces { are literal arguments. (ğŸ”— link)

In literal arguments, Cirodown is not parsed, and the entire argument is considered as text until a corresponding close with the same number of characters. (ğŸ”— link)

Therefore, you cannot have nested content, but it makes it extremely convenient to write Code \c and \C or Mathematics \M and \m. (ğŸ”— link)

For example, a multiline code block with double open and double close square brackets inside can be enclosed in triple square brackets: (ğŸ”— link) 

A literal argument looks like this in Cirodown:

\C[[
\C[
A multiline

cod block.
]
]]

And another paragraph.

The same works for inline code: (ğŸ”— link) 

The program \c[[puts("]");]] is very complex.

Within literal blocks, only one thing can be escaped with backslashes: leading [ or trailing ]. (ğŸ”— link)

The rule is that: (ğŸ”— link)

  • if the first character of a literal argument is a sequence of \, and it is followed by another argument open character (e.g. [), remove the first \ and treat the other characters as regular text
  • if the last character of a literal argument is a \, ignore it and treat the following closing character (e.g. ]) as regular text

<code>\ b</code>

<code>\[ b</code>

\c[[\\\[ b]]
<code>\\[ b</code>

<code>a \</code>

<code>a ]</code>

15.4. The \toplevel implicit macro | â‡‘ toc | â†‘ parent "Developing Cirodown"

Every Cirodown document is implicitly put inside a \toplevel document and: (ğŸ”— link)

  • any optionally given arguments at the very beginning of the document will be treated as arguments of the \toplevel macro
  • anything else will be put inside the content argument of the \toplevel macro

E.g., a Cirodown document that contains: (ğŸ”— link) 

[title=My favorite title]

And now, some content!

is morally equivalent to: (ğŸ”— link) 

\toplevel
[title=My favorite title]
[
And now, some content!
]

In terms of HTML, the \toplevel element corresponds to the <html>, <head>, <header> and <footer> elements of a document. (ğŸ”— link)
