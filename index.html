2. Design goals (toc â†‘)

Cirodown is designed entirely to allow writing complex professional HTML and PDF scientific books, blogs, articles and encyclopedias. (link ðŸ”—)

Cirodown aims to be the ultimate LaTeX "killer", allowing books to be finally published as either HTML or PDF painlessly (LaTeX being only a backend to PDF generation), supporting out of the box feature such as: (link ðŸ”—)

  • references to headers, images, etc. with error checking: never break internal links gain
  • KaTeX server side Mathematics
  • multi file features out of the box so you don't need a separate wrapper like Jekyll to make a multi-page website:
    • Internal cross file references
    • table of contents that crosses input files
    • inotifywait watch rebuild server
    • cross file includes that work as links on multi output file mode, and true includes in single output file mode
    • cross file configuration files to factor out common page parts like headers and footers

It is meant to be both saner and more powerful than Markdown and Asciidoctor. (link ðŸ”—)

The tradeoff for those features is that the language is slightly heavier to read and write. (link ðŸ”—)

It is intended that this will be an acceptable downside as Cirodown will be used primarily large complex content such as books rather than forum posts, and will therefore primarily written either: (link ðŸ”—)

  • in text editors locally, where users have more features than in random browser textareas
  • in a dedicated website that will revolutionize education, and therefore have a good JavaScript editing interface: https://github.com/cirosantilli/write-free-science-books-to-get-famous-website

2.1. Saner (toc â†‘)

To be saner than both Markdown and Asciidoctor, Cirodown has exactly five magic characters, with similar functions as in LaTeX: (link ðŸ”—)

  • \ backslash to start a macro, like LaTeX
  • { and }: left and right square brackets to delimit optional macro arguments
  • [ and ]: left and right curly braces bracket to start an optional arguments

And double blank newlines for paragraphs if you are pedantic. (link ðŸ”—)

We would like to have only square brackets for both optional and mandatory to have even less magic characters, but that would make the language difficult to parse for computer and humans. LaTeX was right for once! (link ðŸ”—)

This produces a very regular syntax that is easy to learn, including doing: (link ðŸ”—)

  • arbitrary nesting of elements
  • adding arbitrary properties to elements

This sanity also makes the end tail learning curve of the endless edge cases found in Markdown and Asciidoctor disappear. (link ðŸ”—)

The language is designed to be philosophically isomorphic to HTML to: (link ðŸ”—)

  • further reduce the learning curve
  • ensure that most of HTML constructs can be reached, including arbitrary nesting

More precisely: (link ðŸ”—)

  • macro names map to tag names, e.g.: \a to <a

  • one of the arguments of macros, maps to the content of the HTML element, and the others map to attributes. (link ðŸ”—)

    E.g., in a link: (link ðŸ”—) 

    \a[http://example.com][Link text]
    (link ðŸ”—)

    the first macro argument: (link ðŸ”—) 

    http://example.com
    (link ðŸ”—)

    maps to the href of <a, and the second macro argument: (link ðŸ”—) 

    Link text
    (link ðŸ”—)

    maps to the internal content of <a>Link text<> (link ðŸ”—)

2.2. More powerful (toc â†‘)

The high sanity of Cirodown, also makes creating new macro extensions extremely easy and intuitive. (link ðŸ”—)

All built-in language features use the exact same API as new extensions, which ensures that the extension API is sane forever. (link ðŸ”—)

Markdown is clearly missing many key features such as block attributes and Internal cross references \x, and has no standardized extension mechanism. (link ðŸ”—)

The "more powerful than Asciidoctor" part is only partially true, since Asciidoctor is very featureful can do basically anything through extensions. (link ðŸ”—)

The difference is mostly that Cirodown is completely and entirely focused on making amazing scientific books, and so will have key features for that application out-of-the box. (link ðŸ”—)

Another advantage over Asciidoctor is that the reference implementation of Cirodown is in JavaScript, and can therefore be used on browser live preview out of the box. Asciidoctor does Transpile to JS with Opal, but who wants to deal with that layer of complexity? (link ðŸ”—)

5. Internal cross references \x (toc â†‘)

Every macro in Cirodown can have an optional id and a title property. (link ðŸ”—)

Those that have a title but no id, get an auto-generated ID from the title: Automatic ID from title. (link ðŸ”—)

For macros that do have an ID, you can write a cross reference to it: (link ðŸ”—) 

\h[2][My title]{id=my-id}

Read this \x[my-id][amazing section].

If the second argument, the content, is not present, it expand to the header title, e.g.: (link ðŸ”—) 

\h[2][My title]{id=my-id}

Read this \x[my-id].

is the same as: (link ðŸ”—) 

\h[2][My title]{id=my-id}

Read this \x[my-id][My title].

5.1. Cross reference style (toc â†‘)

To also show the section auto-generated number as in "Section X.Y My title" we add the optional {style=full} named parameter to the cross reference, for example: (link ðŸ”—) 

\h[2][My title]{id=my-id}

Read this \x[my-id]{style=full}.

5.3. Automatic ID from title (toc â†‘)

If a macro has the title property is present but no explicit id, an ID is created automatically from the title, by applying the following transformations: (link ðŸ”—)

  • convert all of A-Z characters to lowercase
  • convert consecutive sequences of all non a-z0-9 ASCII characters to a single hyphen -. Note that this leaves non-ASCII character untouched.
  • strip leading or trailing hyphens

Note how those rules leave non ASCII Unicode characters untouched, as capitalization and determining if something "is a letter or not" in those cases can be tricky. (link ðŸ”—)

6.4. Includes \include (toc â†‘)

The \include macro achieves the following goals: (link ðŸ”—)

  • show a single header at multiple locations in the table of contents. This makes the TOC into a full blown graph rather than a tree. (link ðŸ”—)

    This can be used to implement a tag entirely within the ToC! (link ðŸ”—)

  • determine where external files will be placed in --single-page mode. (link ðŸ”—)

    When compiling to HTML, we can choose to either: (link ðŸ”—)

    • compile as multiple pages: default, includes show on ToC and render as internal cross references to headers
    • compile as a single page: selected with the cirodown --single-page option
    (link ðŸ”—)

If an external header would be included multiple times in a single output page, the ToC contents are duplicated, but the content itself cannot be because that would lead to ID conflicts and large pages. (link ðŸ”—)

Therefore, in this case you are forced to specify which include will actually expand to the canonical attribute: (link ðŸ”—) 

\include[not-the-readme]{canonical=true}

6.5. Skipping header levels (toc â†‘)

The very first header of a document can be of any level, although we highly recommend your document to start with a \h[1], and to contain exactly just one \h[1], as this has implications such as: (link ðŸ”—)

  • \h[1] is used for the document title: HTML document title
  • \h[1] does not show on the Table of contents \toc

Aftter the initial header however, you must not skip a header level, e.g. the following would give an error because it skips level 3: (link ðŸ”—) 

\h[1][my 1]

\h[2][my 1]

\h[4][my 4]

6.6. The toplevel header (toc â†‘)

If the document has only a single header of the highest level, e.g. like the following has only a single h2: (link ðŸ”—) 

\h[2][My 2]

\h[3][My 3 1]

\h[3][My 3 2]

then this has some magical effects, notably that header IDs won't show that level. For example, the headers would render like: (link ðŸ”—) 

My 2

1. My 3 1

2. My 3 2

rather than: (link ðŸ”—) 

1. My 2

1.2. My 3 1

1.2. My 3 2

This is because in this case, we guess that the h2 is the toplevel (link ðŸ”—)

8.1. Where to store images (toc â†‘)

If you are making a limited repository that will not have a ton of images, then you can get away with simply git tracking your images in the main repository. (link ðŸ”—)

With this setup, no further action is needed. For example, with a file structure of: (link ðŸ”—) 

./README.ciro
./Image_with_a_descriptive_title.jpeg

just use the image from 

README.ciro
as: (link ðŸ”—) 

Here is a nice image:

\Image[Image_with_a_descriptive_title.jpeg]

However, if you are making a huge tutorial, which can have a huge undefined number of images (i.e. any scientific book), then you likely don't want to git track your images in the git repository. (link ðŸ”—)

We recommend the following approach instead. (link ðŸ”—)

Create a separate GitHub repository in addition to the main one containing the text, for example: (link ðŸ”—)

  • ./my-tutorial/
  • ./my-tutorial-media/

The name *-media suffix is not mandatory, but if you use this default, cirodown will handle it for you without any further configuration. (link ðŸ”—)

And then set the media_source_type option in your cirodown.json: (link ðŸ”—) 

{
  media_source_type = "media_repo"
}

Now, simply drop your images into the my-tutorial-media repository and use them just as before, and everything will just work! (link ðŸ”—) 

\Image[Image_with_a_descriptive_title.jpeg]

cirodown will even automatically add and push used images in the my-tutorial-media repository for you during publishing! (link ðŸ”—)

You should then use the following rules inside my-tutorial-media: (link ðŸ”—)

  • give every file a very descriptive and unique name as a full English sentence
  • never ever delete any files, nor change their content, unless it is an improvement in format that does change the information contained of the image TODO link to nice Wikimedia Commons guideline page

This way, even though the repositories are not fully in sync, anyone who clones the latest version of the *-media directory will be able to view any version of the main repository. (link ðŸ”—)

Then, if one day the media repository ever blows up GitHub's limit, you can just migrate the images to another image server that allows arbitrary basenames, e.g. AWS, and just configure your project to use that new media base URL with the media_base_url option. (link ðŸ”—)

The reason why images should be kept in a separate repository is that images are hundreds or thousands of times larger than hand written text. (link ðŸ”—)

Therefore, images could easily fill up the maximum repository size you are allowed: https://webapps.stackexchange.com/questions/45254/file-size-and-storage-limits-on-github#84746 and then what will you do when GitHub comes asking you to reduce the repository size? (link ðŸ”—)

Git LFS is one approach to deal with this, but we feel that it adds too much development overhead. (link ðŸ”—)

9. Code \c and \C (toc â†‘)

Inline code blocks (code blocks that should appear in the middle of a paragraph) are done with lower case c: (link ðŸ”—) 

My \c[inline] code.

renders to: (link ðŸ”—) 

<p>My <code>inline</code> code.</p>

Code blocks (code blocks that should appear outside of paragraphs in their own lines) are done with capital C: (link ðŸ”—) 

A paragraph.

\C[
A block
of code
]

Another paragraph.

renders to: (link ðŸ”—) 

<p>A paragraph.</p>
<pre><code>A block
of code</code></pre>
<p>Another paragraph.</p>

If the content of the code block has many characters that you would need to escape, you will often want to use Literal arguments, which work just like the do for any other argument. For example: (link ðŸ”—) 

A paragraph.

\C[[
And now, some long, long code, with lots
of chars that you would need to escape:
\ [  ] {  }
]]

A paragraph.

Note that the initial newline is skipped automatically in code blocks, just as for any other element, due to: Argument leading newline removal, so you don't have to worry about it. (link ðŸ”—)

The capital vs lower case theme is also used in other elements, for example Mathematics \M and \m. (link ðŸ”—)

The distinction between inline c and block C code blocks is needed because in HTML, pre cannot go inside p. (link ðŸ”—)

We could have chosen to do some magic to differentiate between them, e.g. checking if the block is the only element in a paragraph, but we decided not to do that to keep the language saner. (link ðŸ”—)

13.1. Macro argument syntax (toc â†‘)

13.1.1. Positional vs named arguments (toc â†‘)

Every argument in Cirodown is either positional or named. (link ðŸ”—)

For example, in a header definition with an ID: (link ðŸ”—) 

\h[1][My asdf]{id=asdf qwer}{id_scope=myscope}

we have: (link ðŸ”—)

  • two positional argument: 1 and My asdf. Those are surrounded by and have no name

  • two named arguments: {id=asdf qwer} and {id_scope=myscope}. (link ðŸ”—)

    The first one has name id and the mandatory separator =, followed by the value asdf qwer. (link ðŸ”—)

You can determine if a macro is positional or named by using --help-macros. Its output contains something like: (link ðŸ”—) 

  "h": {
    "name": "h",
    "positional_args": [
      {
        "name": "level"
      },
      {
        "name": "content"
      }
    ],
    "named_args": {
      "id": {
        "name": "id"
      }
      "id_scope": {
        "name": "id_scope"
      }
    },

and so we see that level and content are positional arguments, and id and id_scope are named arguments. (link ðŸ”—)

Generally, positional arguments are few (otherwise it would be hard to know which is which is which), and are almost always used for a given element so tha they save us from typing the name too many times. (link ðŸ”—)

The order of positional arguments must of course be fixed, but named arguments can go anywhere. We can even mix positional and named arguments however we want, although this is not advised for clarity. (link ðŸ”—)

The following are therefore all equivalent: (link ðŸ”—) 

\h[1][My asdf]{id=asdf qwer}{id_scope=myscope}
\h[1][My asdf]{id_scope=myscope}{id=asdf qwer}
\h{id=asdf qwer}{id_scope=myscope}[1][My asdf]
\h{id_scope=myscope}[1]{id=asdf qwer}[My asdf]

Just like named arguments, positional arguments are never mandatory. (link ðŸ”—)

If not given, most positional arguments will default to an empty string. (link ðŸ”—)

However, some positional arguments can have special effects if not given. (link ðŸ”—)

For example, an anchor with the first positional argument present (the URL), but not the second positional argument (the link text) as in: (link ðŸ”—) 

\a[http://example.com]

has the special effect of generating automatic links as in: (link ðŸ”—) 

\a[http://example.com][http://example.com]

See also: Links \a. (link ðŸ”—)

13.1.2. JavaScript interface for arguments (toc â†‘)

The JavaScript interface sees arguments as follows: (link ðŸ”—) 

function macro_name(args)

where args is a dict such that: (link ðŸ”—)

  • optional arguments have the key / value pairs explicitly given on the call

  • mandatory arguments have a key documented by the API, and the value on the call. (link ðŸ”—)

    For example, the link API names its arguments href and text. (link ðŸ”—)

13.1.3. Newlines between arguments (toc â†‘)

The macro name and the first argument, and two consecutive arguments, can be optionally separated by exactly one newline character, e.g.: (link ðŸ”—) 

\h[2]
{id-scope=true}
[Design goals]

is equivalent to: (link ðŸ”—) 

\h[2]{id-scope=true}[Design goals]

and this non-recommended mixed style: (link ðŸ”—) 

\h[2]{id-scope=true}
[Design goals]

This allows to greatly improve the readability of long argument lists by having them one per line. (link ðŸ”—)

13.1.4. Escape characters (toc â†‘)

For non-literal macro arguments, the rule is simple, you must escape all of: (link ðŸ”—)

  • \
  • [ and ]
  • { and }

This is good for short arguments of regular text, but for longer blocks of Code \c and \C or Mathematics \M and \m, you may want to use Literal arguments (link ðŸ”—)

13.1.5. Literal arguments (toc â†‘)

Arguments that are opened with more than one square brackets [ or curly braces { are literal arguments. (link ðŸ”—)

In literal arguments, Cirodown is not parsed, and the entire argument is considered as text until a corresponding close with the same number of characters. (link ðŸ”—)

Therefore, you cannot have nested content, but it makes it extremely convenient to write Code \c and \C or Mathematics \M and \m. (link ðŸ”—)

For example, a multiline code block with double open and double close square brackets inside can be enclosed in triple square brackets: (link ðŸ”—) 

A literal argument looks like this in Cirodown:

\C[[
\C[
A multiline

cod block.
]
]]

And another paragraph.

The same works for inline code: (link ðŸ”—) 

The program \c[[puts("]");]] is very complex.

Within literal blocks, only one thing can be escaped with backslashes: leading [ or trailing ]. (link ðŸ”—)

The rule is that: (link ðŸ”—)

  • if the first character of a literal argument is a sequence of \, and it is followed by another argument open character (e.g. [), remove the first \ and treat the other characters as regular text
  • if the last character of a literal argument is a \, ignore it and treat the following closing character (e.g. ]) as regular text

See the following open input / output: (link ðŸ”—) 

\c[[\ b]]
<code>\ b</code>

\c[[\a b]]
<code>\a b</code>

\c[[\[ b]]
<code>[ b</code>

\c[[\\[ b]]
<code>\[ b</code>

\c[[\\\[ b]]
<code>\\[ b</code>

and close examples: (link ðŸ”—) 

\c[[a \]]
<code>a \</code>

\c[[a \]]]
<code>a ]</code>

\c[[a \\]]]
<code>a \]</code>

13.1.6. Argument leading newline removal (toc â†‘)

If the very first character of an argument is a newline, then that character is ignored. (link ðŸ”—)

For example: (link ðŸ”—) 

\C[[
a

b
]]

generates something like: (link ðŸ”—) 

<pre><code>a (link ðŸ”—)



b
</code></pre>
 (link ðŸ”—)

instead of: (link ðŸ”—) 

<pre><code>
a (link ðŸ”—)



b
</code></pre>
 (link ðŸ”—)

This is extremely convenient to improve the readability of code blocks and similar constructs. (link ðŸ”—)

If you absolutely need an opening newline, just add a second leading or trailing newline to the macro argument, e.g.: (link ðŸ”—) 

\C[[

a

b

]]

13.1.7. Argument automatic indentation removal (toc â†‘)

Inside of a non-literal block, the very first non-whitespace character determines the indentation level of the content. (link ðŸ”—)

This allows seamlessly indenting complex nested content to make it more readable. (link ðŸ”—)

For example, a list with complex content could be written without indentation as: (link ðŸ”—) 

\l[a]
\l[
b

\C[[
And now some code
]]

And a paragraph.
]
\l[c]

but it would be more readable as the equivalent: (link ðŸ”—) 

\l[a]
\l[
  b

  \c[[
  And now some code
  ]]

  And a paragraph.
]
\l[c]

If a something tries to reduce the current indentation level, then that leads to an error: (link ðŸ”—) 

\l[a]
\l[
  b

I'm bad because I have negative indentation.

  Back to good.
]
\l[c]

13.2. Macro capitalization for block vs inline (toc â†‘)

This is just a naming convention without any magic attached to it. It is currently used for macros such as: (link ðŸ”—)

  • Mathematics \M and \m
  • Code \c and \C
  • Comments

We haven't found a sane way of getting rid of that for now, so just for now just force users to explicitly select between them. (link ðŸ”—)

The distinction is required because there are lots of things you cannot put inside paragraphs in HTML. (link ðŸ”—)

14.1.1.2. --publish (toc â†‘)

Cirodown tooling is so amazing that we also take care of the HTML publishing for you. (link ðŸ”—)

14.1.1.2.1. Publish to GitHub Pages (toc â†‘)

If you want to publish your root user page, which appears at `/` (e.g. https://github.com/cirosantilli/cirosantilli.github.io for the user cirosantilli), GitHub annoyingly forces you to use the master branch for the HTML output: (link ðŸ”—)

  • https://github.com/isaacs/github/issues/212
  • https://stackoverflow.com/questions/31439951/how-can-i-use-a-branch-other-than-master-for-user-github-pages

Therefore, you must first place your .ciro source in a different branch and clear up the master branch for the HTML, and only then publish: (link ðŸ”—) 

git checkout -b dev
git branch -D master
git push --delete origin master

git add README.ciro
git commit --message 'some commit message'
cirodown --publish

This will automatically git push both the dev branch for you, and create/update the master branch, which contains the desired HTML output. (link ðŸ”—)

Only changes committed to Git are pushed. (link ðŸ”—)

You then will also want to set your default repository branch to dev in the settings for that repository: https://help.github.com/en/github/administering-a-repository/setting-the-default-branch (link ðŸ”—)

cirodown automatically detects that we are not on the master branch, and therefore guesses that you want to publish GitHub pages on the master branch. (link ðŸ”—)

For any other non-root directory, cirodown sees that you are on the master branch, and automatically publishes on the gh-pages branch, so all you need is: (link ðŸ”—) 

git add README.ciro
git commit --message 'some commit message'
cirodown --publish .

The publishing only happens if the build has no errors. (link ðŸ”—)

When --publish is given, stdin input is not accepted, and so the current directory is built by default, i.e. the following two are equivalent: (link ðŸ”—) 

./cirodown --publish
./cirodown --publish .
14.1.1.3. --publish-commit <commit-message> (toc â†‘)

Like --publish, but also automatically: (link ðŸ”—)

  • git add -u to automatically add change to any files that have been previously git tracked
  • git commit -m <commit-message> to create a new commit with those changes

This allows you to publish your changes live in a single command such as: (link ðŸ”—) 

cirodown --publish-commit 'my amazing change' .

With great power comes great responsibility of course, but who cares! (link ðŸ”—)

15.4. The \toplevel implicit macro (toc â†‘)

Every Cirodown document is implicitly put inside a \toplevel document and: (link ðŸ”—)

  • any optionally given arguments at the very beginning of the document will be treated as arguments of the \toplevel macro
  • anything else will be put inside the content argument of the \toplevel macro

E.g., a Cirodown document that contains: (link ðŸ”—) 

[title=My favorite title]

And now, some content!

is morally equivalent to: (link ðŸ”—) 

\toplevel
[title=My favorite title]
[
And now, some content!
]

In terms of HTML, the \toplevel element corresponds to the <html>, <head>, <header> and <footer> elements of a document. (link ðŸ”—)

15.4.1. HTML document title (toc â†‘)

  • if the title argument of toplevel is given, use that
  • otherwise, if the document has a \h[1], use the title of the first such header
  • otherwise use a dummy value
