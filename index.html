2. Design goals | â‡‘ toc | â†‘ parent "Cirodown"

Cirodown is designed entirely to allow writing complex professional HTML and PDF scientific books, blogs, articles and encyclopedias. (ğŸ”— link)

Cirodown aims to be the ultimate LaTeX "killer", allowing books to be finally published as either HTML or PDF painlessly (LaTeX being only a backend to PDF generation), supporting out of the box feature such as: (ğŸ”— link)

  • references to headers, images, etc. with error checking: never break internal links gain
  • KaTeX server side Mathematics
  • multi file features out of the box so you don't need a separate wrapper like Jekyll to make a multi-page website:
    • Internal cross file references
    • table of contents that crosses input files
    • inotifywait watch rebuild server
    • cross file includes that work as links on multi output file mode, and true includes in single output file mode
    • cross file configuration files to factor out common page parts like headers and footers

It is meant to be both saner and more powerful than Markdown and Asciidoctor. (ğŸ”— link)

The tradeoff for those features is that the language is slightly heavier to read and write. (ğŸ”— link)

It is intended that this will be an acceptable downside as Cirodown will be used primarily large complex content such as books rather than forum posts, and will therefore primarily written either: (ğŸ”— link)

  • in text editors locally, where users have more features than in random browser textareas
  • in a dedicated website that will revolutionize education, and therefore have a good JavaScript editing interface: https://github.com/cirosantilli/write-free-science-books-to-get-famous-website

2.1. Saner | â‡‘ toc | â†‘ parent "Design goals"

To be saner than both Markdown and Asciidoctor, Cirodown has exactly five magic characters, with similar functions as in LaTeX: (ğŸ”— link)

  • \ backslash to start a macro, like LaTeX
  • { and }: left and right square brackets to delimit optional macro arguments
  • [ and ]: left and right curly braces bracket to start an optional arguments

And double blank newlines for paragraphs if you are pedantic. (ğŸ”— link)

We would like to have only square brackets for both optional and mandatory to have even less magic characters, but that would make the language difficult to parse for computer and humans. LaTeX was right for once! (ğŸ”— link)

This produces a very regular syntax that is easy to learn, including doing: (ğŸ”— link)

  • arbitrary nesting of elements
  • adding arbitrary properties to elements

This sanity also makes the end tail learning curve of the endless edge cases found in Markdown and Asciidoctor disappear. (ğŸ”— link)

The language is designed to be philosophically isomorphic to HTML to: (ğŸ”— link)

  • further reduce the learning curve
  • ensure that most of HTML constructs can be reached, including arbitrary nesting

More precisely: (ğŸ”— link)

  • macro names map to tag names, e.g.: \a to <a

  • one of the arguments of macros, maps to the content of the HTML element, and the others map to attributes. (ğŸ”— link)

    E.g., in a link: (ğŸ”— link) 

    \a[http://example.com][Link text]
    (ğŸ”— link)

    the first macro argument: (ğŸ”— link) 

    http://example.com
    (ğŸ”— link)

    maps to the href of <a, and the second macro argument: (ğŸ”— link) 

    Link text
    (ğŸ”— link)

    maps to the internal content of <a>Link text<> (ğŸ”— link)

2.2. More powerful | â‡‘ toc | â†‘ parent "Design goals"

The high sanity of Cirodown, also makes creating new macro extensions extremely easy and intuitive. (ğŸ”— link)

All built-in language features use the exact same API as new extensions, which ensures that the extension API is sane forever. (ğŸ”— link)

Markdown is clearly missing many key features such as block attributes and Internal cross references \x, and has no standardized extension mechanism. (ğŸ”— link)

The "more powerful than Asciidoctor" part is only partially true, since Asciidoctor is very featureful can do basically anything through extensions. (ğŸ”— link)

The difference is mostly that Cirodown is completely and entirely focused on making amazing scientific books, and so will have key features for that application out-of-the box. (ğŸ”— link)

Another advantage over Asciidoctor is that the reference implementation of Cirodown is in JavaScript, and can therefore be used on browser live preview out of the box. Asciidoctor does Transpile to JS with Opal, but who wants to deal with that layer of complexity? (ğŸ”— link)

5. Internal cross references \x | â‡‘ toc | â†‘ parent "Cirodown"

Every macro in Cirodown can have an optional id and a title property. (ğŸ”— link)

Those that have a title but no id, get an auto-generated ID from the title: Automatic ID from title. (ğŸ”— link)

For macros that do have an ID, you can write a cross reference to it: (ğŸ”— link)

5.1. x child argument | â‡‘ toc | â†‘ parent "Internal cross references \x"

Setting child=1 on a cross reference to a header as in: (ğŸ”— link) 

\xref[my-header]{child=1}

has the following effects: (ğŸ”— link)

  • add the current header to the list of extra parents of the child

This allows a section to have multiple parents, e.g. to include it into multiple categories. (ğŸ”— link)

For example: (ğŸ”— link) 

\h[1][Animals]

\h[2][Mammals]

\h[3][Dog]

\h[3][Cat]

\h[2][Cute animals]

Cute animals:

\l[\xref[dog]{child=1}]
\l[\xref[cat]]

would render something like: (ğŸ”— link) 

h1 Animals

h2 Mammals

h3 Dog (parent section: Mammals)
(other parent sections: Cute animals)

h3 Cat (parent section: Mammals)

\h[2][Cute animals]

\l[\xref[dog]]
\l[\xref[cat]]

so note how "Dog" has a list of extra parents including "Cute animals", but Cat does not, due to the child=1. (ğŸ”— link)

This property does not affect how the \toc is rendered. We could insert elements sections there multiple times, but it has the downside that browser Ctrl + F searches would hit the same thing multiple times on the table of contents, which might make finding things harder. (ğŸ”— link) 

\h[2][My title]{id=my-id}

Read this \x[my-id][amazing section].

If the second argument, the content, is not present, it expand to the header title, e.g.: (ğŸ”— link) 

\h[2][My title]{id=my-id}

Read this \x[my-id].

is the same as: (ğŸ”— link) 

\h[2][My title]{id=my-id}

Read this \x[my-id][My title].

5.2. Cross reference style | â‡‘ toc | â†‘ parent "Internal cross references \x"

To also show the section auto-generated number as in "Section X.Y My title" we add the optional {style=full} named parameter to the cross reference, for example: (ğŸ”— link) 

\h[2][My title]{id=my-id}

Read this \x[my-id]{style=full}.

Example: Section 5.2. "Cross reference style". (ğŸ”— link)

5.4. Automatic ID from title | â‡‘ toc | â†‘ parent "Internal cross references \x"

If a macro has the title property is present but no explicit id, an ID is created automatically from the title, by applying the following transformations: (ğŸ”— link)

  • convert consecutive sequences of all non a-z0-9 ASCII characters to a single hyphen -. Note that this leaves non-ASCII character untouched.
  • strip leading or trailing hyphens

Note how those rules leave non ASCII Unicode characters untouched, as capitalization and determining if something "is a letter or not" in those cases can be tricky. (ğŸ”— link)

6.3. Header explicit levels vs nesting design choice | â‡‘ toc | â†‘ parent "Headers \h"

Arguably, the language would be even saner if we did: (ğŸ”— link) 

\h[My h1][

Paragraph.

\h[My h2][]
]

rather than having explicit levels. (ğŸ”— link)

But we chose not to do it like most markups available because it leads to too many nestiing levels, and hard to determine where you are without tooling. (ğŸ”— link)

6.4. Includes \include | â‡‘ toc | â†‘ parent "Headers \h"

The \include macro achieves the following goals: (ğŸ”— link)

  • show a single header at multiple locations in the table of contents. This makes the TOC into a full blown graph rather than a tree. (ğŸ”— link)

    This can be used to implement a tag entirely within the ToC! (ğŸ”— link)

  • determine where external files will be placed in --single-page mode. (ğŸ”— link)

    When compiling to HTML, we can choose to either: (ğŸ”— link)

    • compile as multiple pages: default, includes show on ToC and render as internal cross references to headers
    • compile as a single page: selected with the cirodown --single-page option
    (ğŸ”— link)

If an external header would be included multiple times in a single output page, the ToC contents are duplicated, but the content itself cannot be because that would lead to ID conflicts and large pages. (ğŸ”— link)

Therefore, in this case you are forced to specify which include will actually expand to the canonical attribute: (ğŸ”— link) 

\include[not-the-readme]{canonical=true}

6.5. Skipping header levels | â‡‘ toc | â†‘ parent "Headers \h"

The very first header of a document can be of any level, although we highly recommend your document to start with a \h[1], and to contain exactly just one \h[1], as this has implications such as: (ğŸ”— link)

  • \h[1] is used for the document title: HTML document title
  • \h[1] does not show on the Table of contents \toc

Aftter the initial header however, you must not skip a header level, e.g. the following would give an error because it skips level 3: (ğŸ”— link) 

\h[1][my 1]

\h[2][my 1]

\h[4][my 4]

6.6. The toplevel header | â‡‘ toc | â†‘ parent "Headers \h"

If the document has only a single header of the highest level, e.g. like the following has only a single h2: (ğŸ”— link) 

\h[2][My 2]

\h[3][My 3 1]

\h[3][My 3 2]

then this has some magical effects, notably that header IDs won't show that level. For example, the headers would render like: (ğŸ”— link) 

My 2

1. My 3 1

2. My 3 2

rather than: (ğŸ”— link) 

1. My 2

1.2. My 3 1

1.2. My 3 2

This is because in this case, we guess that the h2 is the toplevel (ğŸ”— link)

8.1. Where to store images | â‡‘ toc | â†‘ parent "Images \image"

If you are making a limited repository that will not have a ton of images, then you can get away with simply git tracking your images in the main repository. (ğŸ”— link)

With this setup, no further action is needed. For example, with a file structure of: (ğŸ”— link) 

./README.ciro
./Image_with_a_descriptive_title.jpeg

just use the image from 

README.ciro
as: (ğŸ”— link) 

Here is a nice image:

\Image[Image_with_a_descriptive_title.jpeg]

However, if you are making a huge tutorial, which can have a huge undefined number of images (i.e. any scientific book), then you likely don't want to git track your images in the git repository. (ğŸ”— link)

We recommend the following approach instead. (ğŸ”— link)

Create a separate GitHub repository in addition to the main one containing the text, for example: (ğŸ”— link)

  • ./my-tutorial/
  • ./my-tutorial-media/

The name *-media suffix is not mandatory, but if you use this default, cirodown will handle it for you without any further configuration. (ğŸ”— link)

And then set the media_source_type option in your cirodown.json: (ğŸ”— link) 

{

  • never ever delete any files, nor change their content, unless it is an improvement in format that does change the information contained of the image TODO link to nice Wikimedia Commons guideline page

13.1. Macro argument syntax | â‡‘ toc | â†‘ parent "Cirodown syntax"

13.1.1. Positional vs named arguments | â‡‘ toc | â†‘ parent "Macro argument syntax"

Every argument in Cirodown is either positional or named. (ğŸ”— link)

For example, in a header definition with an ID: (ğŸ”— link) 

\h[1][My asdf]{id=asdf qwer}{id_scope=myscope}

we have: (ğŸ”— link)

  • two positional argument: 1 and My asdf. Those are surrounded by and have no name

  • two named arguments: {id=asdf qwer} and {id_scope=myscope}. (ğŸ”— link)

    The first one has name id and the mandatory separator =, followed by the value asdf qwer. (ğŸ”— link)

You can determine if a macro is positional or named by using --help-macros. Its output contains something like: (ğŸ”— link) 

  "h": {
    "name": "h",
    "positional_args": [
      {
        "name": "level"
      },
      {
        "name": "content"
      }
    ],
    "named_args": {
      "id": {
        "name": "id"
      }
      "id_scope": {
        "name": "id_scope"
      }
    },

and so we see that level and content are positional arguments, and id and id_scope are named arguments. (ğŸ”— link)

Generally, positional arguments are few (otherwise it would be hard to know which is which is which), and are almost always used for a given element so tha they save us from typing the name too many times. (ğŸ”— link)

The order of positional arguments must of course be fixed, but named arguments can go anywhere. We can even mix positional and named arguments however we want, although this is not advised for clarity. (ğŸ”— link)

The following are therefore all equivalent: (ğŸ”— link) 

\h[1][My asdf]{id=asdf qwer}{id_scope=myscope}
\h[1][My asdf]{id_scope=myscope}{id=asdf qwer}
\h{id=asdf qwer}{id_scope=myscope}[1][My asdf]
\h{id_scope=myscope}[1]{id=asdf qwer}[My asdf]

Just like named arguments, positional arguments are never mandatory. (ğŸ”— link)

If not given, most positional arguments will default to an empty string. (ğŸ”— link)

However, some positional arguments can have special effects if not given. (ğŸ”— link)

For example, an anchor with the first positional argument present (the URL), but not the second positional argument (the link text) as in: (ğŸ”— link) 

\a[http://example.com]

has the special effect of generating automatic links as in: (ğŸ”— link) 

\a[http://example.com][http://example.com]

See also: Links \a. (ğŸ”— link)

13.1.2. JavaScript interface for arguments | â‡‘ toc | â†‘ parent "Macro argument syntax"

The JavaScript interface sees arguments as follows: (ğŸ”— link) 

function macro_name(args)

where args is a dict such that: (ğŸ”— link)

  • optional arguments have the key / value pairs explicitly given on the call

  • mandatory arguments have a key documented by the API, and the value on the call. (ğŸ”— link)

    For example, the link API names its arguments href and text. (ğŸ”— link)

13.1.3. Newlines between arguments | â‡‘ toc | â†‘ parent "Macro argument syntax"

The macro name and the first argument, and two consecutive arguments, can be optionally separated by exactly one newline character, e.g.: (ğŸ”— link)

{id-scope=true}
[Design goals]

is equivalent to: (ğŸ”— link)

and this non-recommended mixed style: (ğŸ”— link)

[Design goals]

This allows to greatly improve the readability of long argument lists by having them one per line. (ğŸ”— link)

13.1.4. Escape characters | â‡‘ toc | â†‘ parent "Macro argument syntax"

For non-literal macro arguments, the rule is simple, you must escape all of: (ğŸ”— link)

  • \
  • [ and ]
  • { and }

This is good for short arguments of regular text, but for longer blocks of Code \c and \C or Mathematics \M and \m, you may want to use Literal arguments (ğŸ”— link)

13.1.5. Literal arguments | â‡‘ toc | â†‘ parent "Macro argument syntax"

Arguments that are opened with more than one square brackets [ or curly braces { are literal arguments. (ğŸ”— link)

In literal arguments, Cirodown is not parsed, and the entire argument is considered as text until a corresponding close with the same number of characters. (ğŸ”— link)

Therefore, you cannot have nested content, but it makes it extremely convenient to write Code \c and \C or Mathematics \M and \m. (ğŸ”— link)

For example, a multiline code block with double open and double close square brackets inside can be enclosed in triple square brackets: (ğŸ”— link) 

A literal argument looks like this in Cirodown:

\C[[
\C[
A multiline

cod block.
]
]]

And another paragraph.

The same works for inline code: (ğŸ”— link) 

The program \c[[puts("]");]] is very complex.

Within literal blocks, only one thing can be escaped with backslashes: leading [ or trailing ]. (ğŸ”— link)

The rule is that: (ğŸ”— link)

  • if the first character of a literal argument is a sequence of \, and it is followed by another argument open character (e.g. [), remove the first \ and treat the other characters as regular text
  • if the last character of a literal argument is a \, ignore it and treat the following closing character (e.g. ]) as regular text

See the following open input / output: (ğŸ”— link) 

\c[[\ b]]
<code>\ b</code>

\c[[\a b]]
<code>\a b</code>

\c[[\[ b]]
<code>[ b</code>

\c[[\\[ b]]
<code>\[ b</code>

\c[[\\\[ b]]
<code>\\[ b</code>

and close examples: (ğŸ”— link) 

\c[[a \]]
<code>a \</code>

\c[[a \]]]
<code>a ]</code>

\c[[a \\]]]
<code>a \]</code>

13.1.6. Argument leading newline removal | â‡‘ toc | â†‘ parent "Macro argument syntax"

If the very first character of an argument is a newline, then that character is ignored. (ğŸ”— link)

For example: (ğŸ”— link) 

\C[[
a

b
]]

generates something like: (ğŸ”— link) 

<pre><code>a (ğŸ”— link)



b
</code></pre>
 (ğŸ”— link)

instead of: (ğŸ”— link) 

<pre><code>
a (ğŸ”— link)



b
</code></pre>
 (ğŸ”— link)

This is extremely convenient to improve the readability of code blocks and similar constructs. (ğŸ”— link)

If you absolutely need an opening newline, just add a second leading or trailing newline to the macro argument, e.g.: (ğŸ”— link) 

\C[[

a

b

]]

13.1.7. Argument automatic indentation removal | â‡‘ toc | â†‘ parent "Macro argument syntax"

Inside of a non-literal block, the very first non-whitespace character determines the indentation level of the content. (ğŸ”— link)

This allows seamlessly indenting complex nested content to make it more readable. (ğŸ”— link)

For example, a list with complex content could be written without indentation as: (ğŸ”— link) 

\l[a]
\l[
b

\C[[
And now some code
]]

And a paragraph.
]
\l[c]

but it would be more readable as the equivalent: (ğŸ”— link) 

\l[a]
\l[
  b

  \c[[
  And now some code
  ]]

  And a paragraph.
]
\l[c]

If a something tries to reduce the current indentation level, then that leads to an error: (ğŸ”— link) 

\l[a]
\l[
  b

I'm bad because I have negative indentation.

  Back to good.
]
\l[c]

14.1.1.2.1. Publish to GitHub Pages | â‡‘ toc | â†‘ parent "--publish"

If you want to publish your root user page, which appears at `/` (e.g. https://github.com/cirosantilli/cirosantilli.github.io for the user cirosantilli), GitHub annoyingly forces you to use the master branch for the HTML output: (ğŸ”— link)

  • https://stackoverflow.com/questions/31439951/how-can-i-use-a-branch-other-than-master-for-user-github-pages

Therefore, you must first place your .ciro source in a different branch and clear up the master branch for the HTML, and only then publish: (ğŸ”— link) 

git checkout -b dev
git branch -D master
git push --delete origin master

git add README.ciro
git commit --message 'some commit message'
cirodown --publish

This will automatically git push both the dev branch for you, and create/update the master branch, which contains the desired HTML output. (ğŸ”— link)

Only changes committed to Git are pushed. (ğŸ”— link)

You then will also want to set your default repository branch to dev in the settings for that repository: https://help.github.com/en/github/administering-a-repository/setting-the-default-branch (ğŸ”— link)

cirodown automatically detects that we are not on the master branch, and therefore guesses that you want to publish GitHub pages on the master branch. (ğŸ”— link)

For any other non-root directory, cirodown sees that you are on the master branch, and automatically publishes on the gh-pages branch, so all you need is: (ğŸ”— link) 

git add README.ciro
git commit --message 'some commit message'
cirodown --publish .

The publishing only happens if the build has no errors. (ğŸ”— link)

When --publish is given, stdin input is not accepted, and so the current directory is built by default, i.e. the following two are equivalent: (ğŸ”— link) 

./cirodown --publish
./cirodown --publish .
